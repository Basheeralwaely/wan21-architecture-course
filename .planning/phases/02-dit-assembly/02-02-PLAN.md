- forward(self, x, timestep, context, seq_len=None, clip_feature=None, y=None, use_gradient_checkpointing=False):
  Step 1: t = time_embedding(sinusoidal_embedding_1d(timestep, freq_dim))  -> [B, dim]
  Step 2: t_mod = time_projection(t).unflatten(1, (6, dim))              -> [B, 6, dim]
  Step 3: context = text_embedding(context)                               -> [B, L, dim]
  Step 4: x = patchify(x)                                                 -> [B, S, dim]
  Step 5: freqs = cat(f_freqs, h_freqs, w_freqs).reshape(S, 1, head_dim//2)
  Step 6: for block in blocks: x = block(x, context, t_mod, freqs)       -> [B, S, dim]
  Step 7: x = head(x, t)                                                  -> [B, S, out_dim*prod(ps)]
  Step 8: x = unpatchify(x)                                               -> [B, out_dim, F, H, W]

Cell 12 (markdown): "### Why Gradient Checkpointing Matters for LoRA Training" prose connecting to NB-06's LoRA parameter count. With 30 blocks of ~51M params each storing activations, peak VRAM is prohibitive. Checkpointing stores only the input/output per block and recomputes intermediate activations during backward pass. Trades ~25% extra compute for significant VRAM reduction.
